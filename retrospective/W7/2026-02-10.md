# 리뷰
## 최종 프로젝트 API 아키텍처 설계
### 아키텍처 회의
* 강사님 피드백처럼 완벽한 설계보다는 일단 다 쓰는(EC2 + Spark + Lambda + RDS + S3) 버전으로 해보고 최적화는 나중에 하기로 함
* 그래도 내일 현직자 멘토링 받을 때 아키텍처 관련 피드백을 받으려면 최종 아키텍처가 어떻게 나올 수 있을지 variation 정도는 생각해보기로 함

* 팀원 각자가 생각한 아키텍처를 공유하고 어떤 variation이 가능할지 토론하면서 4가지 버전으로 정리함
    * EC2 micro가 RAM이 1GB밖에 안돼서 Spark가 안 돌아갈 가능성 -> pandas 기반 처리 또는 EMR 사용도 열어 두기 (현실적인 제한으로 인한 대안)
    * API 서버용 EC2를 안 쓰고 API Gateway + lambda 조합으로 가는 것도 고려 (EC2 비용 절감 버전)
    * 고객이 정해진 쿼리만 요청하고 결과 데이터가 작으면, ETL 할 때 결과를 미리 저장해뒀다가 RDS 안 쓰고 Lambda에서 바로 반환하는 구조도 가능하다고 판단 (RDS 비용 절감 버전)

### ETL 로직 구체화
* '어떻게 해결'을 구현하기 위해 식당의 부설주차장 위치를 알아내야 했음 -> 30분 타임박싱
* 타임박싱 끝날 때까지 공개 데이터를 못 찾아서 타임박싱 10분 연장
* 공개 데이터가 없다고 판단하고 추정 로직을 고민함 (10분 더 연장, 총 20분 고민)
    * 식당 필지에 인접한 필지 중, 건물이 아니고, 주차장 용도로 사용 가능한 지목이면 부설주차장으로 추정
    * 인접한 필지를 어떻게 판단할 수 있는지 api 찾아봄
* 타임 박싱이 끝나고 각자 알아본 내용을 공유하는데, 팀원이 좋은 아이디어를 제안함
    * 식당 인접 토지의 소유변경일자가 식당과 동일하면 식당 소유자가 함께 구매했을 가능성이 크기 때문에 해당 토지를 부설주차장으로 추정하자는 아이디어
    * 샘플로 식당 몇 곳만 찍어서 소유변경일자와 실소유주를 확인함 -> 실제로 소유변경일자가 같으면 소유자가 같고 부설주차장이었음
* 잘 작동할 것이라고 찍고 일단 구현하자고 의사결정!
    * 나는 공개데이터 있는지만 찾아보다가 30분 끝나서야 추정 로직을 고민했는데 그동안 핵심 데이터 조인해보고 인사이트 찾고 계셨던게 너무 대단하게 느껴졌음
    * 있는 데이터도 안 찍어보고 공개 데이터만 찾으려고 했던 것도 지적 게으름인 것 같음... 반성하자!!

### 프리토타입 작성
* api request, response 구조를 고민하면서 고객이 실제로 보고 싶어 할 데이터가 뭘지 고민함
    * 설계 과정에서 추가할 파라미터도 정하고, 필터를 범주형으로 할지 범위형으로 할지도 정하면서 '어떤 문제'를 조금씩 바꿈

### 새로운 아이디어 구상
* 어제 회고에 썼던 "리스크 줄이기" 관점으로 강사님이 피드백에서 언급하셨던 "자율주행 대기 장소" 문제 재검토
* 현대자동차의 입장에서 어떤 사업을 하게 될까 생각하다가 자율주행 택시나 화물 운송쪽으로 확장할 것 같다는 생각을 함
* 자율주행 택시/화물 트럭 대기 문제 제시
    * 대기 위치가 멀면 빈 차로 운행하는 거리가 늘어남 -> 수익 손실
    * 효율적인 대기 위치는 어떻게 구할 수 있을까?
    * 택시는 유동인구, 화물 트럭은 물류센터 위치에 따라 문제가 달라짐
    * 구체적인 '어떤 문제' 및 '어떻게 해결'은 내일 정리하기로 결정
