* 중요한 건 이론이 아니라 무엇을 왜 선택해야 하는지 의사결정의 문제임

    * 이런 것들은 problem에서 결정됨 (누구의, 어떤 문제)

* Spark 아키텍처

        * 용어 자체가 비슷하고 옵션에 모드까지 같이 생각하니까 헷갈려서 더 정리가 필요할 것 같음

## W4M1
* root가 아닌 spark 유저 생성해서 권한 부여
    * 디렉토리 생성, 파일 write 과정에서 권한 문제가 발생해서 해결
    * 유저 생성 하나 추가하는 걸로도 다양한 이슈가 발생할 수 있을 것 같음
        * 많이 적용해보고, 해결해보면서 익숙해지기
* Docker Compose를 사용해 Spark Standalone 클러스터 구성함

## 팀 회의
* '시간과 정보 부족' 이라는 말이 너무 추상적이라 문제가 뭔지를 흐리고 있다는 의견이 나옴
    * 상황을 더 디테일하게 보충하고 문제는 핵심만 남게 덜어내는 방향으로 수정하기로 함
    * 국내, 해외 변호사 홈페이지를 직접 검색해보면서 진짜 문제가 뭘지 고민하는 시간을 가짐 
* 고통이 더 큰 타겟이 있을지 내일 아침까지 더 생각해보기로 함

1. 매주 금요일 퇴근 전 10분 동안 일주일 동안 쓴 회고를 다시 읽고 리뷰하기

## 노력 필요
1. 모든 작업 최대 30분씩 타임박싱 하고, 연장은 최대 1번만 하기 (작업 = 해결 할 작은 문제 1개 e.g., 도커 파일 작성)
2. 데드라인보다 하루 더 일찍 끝내는 것을 목표로 스케줄링 하기
3. 미션 1개 해결 이후 10분 동안 해당 미션 리뷰하기
