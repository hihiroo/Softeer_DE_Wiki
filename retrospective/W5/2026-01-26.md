# 리뷰
## 직무교육
### Lineage와 action 기반 실행 방식
* Spark는 lineage를 기반으로 DAG를 구성하고, action이 실행되는 시점에만 실제 계산이 수행되는 lazy evaluation 구조라는 걸 학습함
    * 학부 때 spark 실습을 하면서 실행시간이 이상하게 나왔던 원인이 lazy evaluation 때문인 것 같아서 신기했고, 이런 설계 지식 없이 코드를 짜고 디버깅하는건 위험할 수 있겠다고 느낌
* Stage가 만들어지는 구조를 보면서, 병렬화를 극대화하려면 액션 위치까지 고려해야 된다는 생각이 들었음

### Spark Shuffle 과정 조사
* 직무 교육 이후에 '셔플 과정에서 메모리가 터지면 어떡하지?' 라는 궁금증이 생겨서 추가로 조사해봄
* 각 파티션으로 보낼 데이터를 각각의 파일로 만들어서 disk에 쓰고, 이 파일들을 읽어서 처리하는데, 이때 메모리가 부족하면 디스크로 spill될 수 있다는 것을 배움
* 셔플할 데이터를 파일로 쓰는 과정에서 disk I/O가 일어나고, 이 파일들을 서로 다른 파티션이 주고 받는 과정에서 네트워크 I/O가 일어나고, 파일을 읽는 과정에서 또 disk I/O가 발생할 수 있다는 점에서 왜 셔플이 심각한 병목으로 이어지는지 이해함
* 종이에 disk, 메모리, 파티션, 노드 직접 그리면서 데이터 흐름 시각화 하니까 이해가 더 빠르게 되었음

## 최종 프로젝트 팀 회의
### 그라운드룰 / 일정
* 이전 팀에서의 경험을 공유하면서 **잘 됐던 방식**을 뽑아 그라운드룰로 정함
* 일정은 투트랙 방식을 기본으로 계획하되, 개인 미션과 아이디에이션, 개발을 어떻게 병렬 배치할 지 논의함
    * 수요일까지는 개인 미션 부담이 있을 것 같아 아이디에이션만 진행하기로 함
    * 이후에는 아이디에이션 + 수요일까지 나왔던 아이디어 중 가장 좋은 아이디어 뽑아서 미니 프로젝트 진행
* 멘토링 전까지 아키텍처 + UI 포함 프리토타입 완성하는 것을 목표로 잡음

### 아이디어 회의
* **마인드맵**을 그리면서 차량과 관련된 어떤 소재가 있을지 먼저 전체적으로 정리
* 서로 그린 마인드맵을 공유하면서 연결 아이디어가 있는지 찾아봄
* 소재는 많이 나왔지만, '문제'를 못 찾았음
    * 차량 도메인에서 팀원들이 실제로 겪은 문제 경험이 부족한 것 같음
* 주변 사람들에게 실제로 겪은 문제상황을 물어보거나 자료조사를 통해 현실의 pain point를 먼저 확보하기로 함
* 마인드맵은 내일 통합 정리하며 생각 못 한 소재가 있을지 추가 탐색할 예정
